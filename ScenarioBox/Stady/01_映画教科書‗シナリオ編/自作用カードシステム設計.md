# 自作用カードシステム設計

## 目的

執筆中に散らばるアイデア、イベント、シーン、キャラクター、用語をカード化し、作品をまたいで検索・再利用できるようにする。

特に重視すること:

- Androidでも書きやすい
- 前に書いた情報へすぐ戻れる
- 作品タイトルをまたいだアイデア移動ができる
- カード表示、タグ表示、アウトライン表示を切り替えられる
- `Xa -> Xb` の状態変化でイベントを管理できる

## カード種別

| 種別 | 用途 |
|---|---|
| idea | まだ物語に接続していない断片 |
| event | 状態変化を持つ出来事 |
| scene | 場所と時間で区切られたシーン |
| episode | 複数シーンを束ねる単位 |
| character | 人物の役割・目的・障害・対策 |
| term | 用語、技法、制作メモ |
| tarot_keyword | 抽象キーワード、占い・象徴カード |
| analysis | 映画分析、参考作品メモ |

### チェック観点

- `X` は誰か
- `a` は何か
- `b` は何か
- なぜ変化するのか
- 何が変化を邪魔するのか
- その出来事は物語全体のどこに置かれるのか

### 書くこと

- 場所
- 時間
- 登場人物
- 見える行動
- そのシーンで変わること

### 書きすぎないこと

- 細かすぎるカット割
- 俳優の演技指定
- 内面の説明
- 日常会話の丸写し

## 表示モード

| 表示 | 目的 |
|---|---|
| カード表示 | 断片を眺めて並べ替える |
| タグ表示 | 類似アイデアを横断検索する |
| アウトライン表示 | 全体から細部へ移動する |
| 状態変化表示 | `Xa -> Xb` の流れを確認する |
| 分析表示 | 参考映画の構造を確認する |

## 最小実装案

1. Markdown内にYAML付きカードブロックを書く
2. `type` と `tags` でカード一覧を生成する
3. `work_title` で作品ごとに絞り込む
4. `related_cards` でカード間ジャンプを作る
5. 将来的に付箋UIやカードUIへ展開する

## 例

```yaml
id: event-001
type: event
title: 主人公が秘密を知る
work_title: サンプル作品
character_x: 主人公
before_state_a: 真実を知らない
after_state_b: 父の死の理由を知る
change: 無知から認識へ
motivation: 家族の謎を解きたい
obstacle: 周囲が真実を隠している
countermeasure: 古い手紙を調べる
result: 次のシーンで対決へ向かう
tags:
  - 状態変化
  - 家族
  - 真実
related_cards:
  - character-001
  - scene-003
```
