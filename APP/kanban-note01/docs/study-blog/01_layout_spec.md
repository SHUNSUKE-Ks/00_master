# Study Blog レイアウト書

## 1. 画面目的

必要な技術情報へ短時間で到達し、出先のAndroid端末で確認できることを優先する。
記事を順番に読む一般ブログではなく、社内WIKI、技術ポータル、AI指示Card集に近い画面とする。

## 2. 基本動線

```text
Note_Storyメニュー
  → Study
    → Module一覧
      → Card一覧
        → Card本文
          → 関連Card / 用語 / コード参照
```

戻る階層は必ず次の順番にする。

```text
Card本文 → Card一覧 → Module一覧 → Note_Storyメニュー
```

選択中のModuleとCardは保持し、空の中間画面を作らない。

共通HeaderにはGlobal InBox送信ボタンを常設する。Androidでは飛行機アイコンのみ、PCでは`InBox`ラベルを併記する。

## 3. Android縦画面

### 3.1 Module一覧

```text
┌─────────────────────────┐
│ Note_Story   Study      │
├─────────────────────────┤
│ 検索...          Filter │
├─────────────────────────┤
│ 状態駆動                │
│ State / Action / View   │
│ 6 Cards      学習中     │
├─────────────────────────┤
│ Notebook構造            │
│ Page / Log / Character  │
│ 8 Cards      検証済     │
└─────────────────────────┘
```

- 1カラム表示。
- Module名を第一情報にする。
- 説明は2行まで。
- Card数、状態、主要Tagだけを表示する。
- カード内に操作説明文を置かない。

### 3.2 Card一覧

```text
┌─────────────────────────┐
│ ←  状態駆動             │
│ 概要 > State > 派生     │
├─────────────────────────┤
│ [概念] Solidの状態      │
│ 状態を正本にする考え方  │
│ solidjs  state  基礎    │
├─────────────────────────┤
│ [実装] createStore      │
│ Noteでの利用箇所        │
│ store  action  code     │
└─────────────────────────┘
```

- Module内の学習順を初期並び順にする。
- 必須Card、任意Card、後続Cardを区別する。
- フィルター後も現在のModuleを保持する。

### 3.3 Card本文

```text
┌─────────────────────────┐
│ ← Card一覧       ☆ 索引 │
├─────────────────────────┤
│ 状態駆動 > 派生状態     │  固定パンくず
├─────────────────────────┤
│ 派生状態                │  H1
│ 目的 / 対象版 / 状態    │
│                         │
│ ■ 要点                  │  H2
│ ・状態から計算する      │  H3
│                         │
│ ┌ Code ──────────────┐ │
│ │ // 状態から導出     │ │
│ │ const page = () =>  │ │
│ └─────────────────────┘ │
│                         │
│ 関連: createMemo →      │
│                    ┌─┐  │
│                    │目│  │  右端固定Button
│                    │次│  │
│                    └─┘  │
└─────────────────────────┘
```

- パンくずは画面上部へ追従させる。
- `Module > Card > 現在見出し`を表示する。
- H2/H3を目次から選ぶと本文位置へジャンプする。
- コードは横スクロール可能にし、本文全体を横へ広げない。
- 関連語句は本文中リンクと末尾の関連一覧の両方から移動できる。

### 3.4 スライドイン目次

パンくずとは別に、画面を占有しない開閉式の縦型目次を持つ。

```text
通常表示                 目次を開いた状態
┌──────────────┐       ┌──────────────┐
│ Card本文   ┌─┐│       │暗いOverlay ┌───────┐│
│            │目││       │            │ 目次  ││
│            │次││  →    │            │■ H2  ││
│            └─┘│       │            │ ・H3 ││
│              │       │            │■ H2  ││
└──────────────┘       └────────────┴───────┘
```

- `目次`Buttonは画面右端中央付近へ固定する。
- Buttonは縦長の長方形とし、文字を縦方向に`目次`と表示する。
- 本文幅を常時狭めず、開いた時だけ右側からパネルをスライドインする。
- パネル外へ半透明Overlayを表示する。
- Overlay、閉じるButton、Androidの戻る操作で閉じられる。
- H2は`■`、H3は`・`で階層表示する。
- 見出しをタップすると該当位置までスクロールし、Androidでは目次を自動で閉じる。
- 現在読んでいる見出しを目次内で強調する。
- スクロール位置と追従パンくずを同じ見出し判定から更新する。
- 目次のスクロール位置は、現在見出しが見える位置へ追従する。

### 3.5 Copy操作

コード、JSON、Schemaは右上にCopyアイコンを持つ。

```text
┌ TypeScript ─────────── [Copy] ┐
│ const selected = () => ...    │
└───────────────────────────────┘
```

- アイコンはコード枠の右上へ固定する。
- タップ領域は最低36px程度を確保する。
- コピー対象は表示中のコード全体とする。
- 成功時は短時間`Copied`表示へ切り替える。
- 失敗時は本文を失わず、`コピー失敗`を表示する。
- コード言語名を左上へ表示できる。
- 横スクロールしてもCopyアイコンは枠内右上に残す。

## 4. PC画面

```text
┌ Module一覧 ┬ Card一覧 ┬ Card本文 ─────────┐
│            │          │ パンくず           │
│            │          │ 目次 / 本文 / Code │
└────────────┴──────────┴───────────────────┘
```

- PCは3ペインまで許可する。
- 主体はCard本文であり、左右ペインは折り畳める。
- Androidと情報構造を変えず、表示密度だけを増やす。

## 5. 文字基準

| 要素 | Android基準 |
| --- | --- |
| 本文 | 16px、行間1.8 |
| 補足 | 13px、行間1.6 |
| H1 | 24px、太字 |
| H2 | 19px、太字 |
| H3 | 16px、太字 |
| Code | 13px、行間1.6 |
| 本文余白 | 左右16px |

初期フォントは端末のシステムフォントを使う。
独自フォントはStudy Cardで比較・検証後に導入する。

## 6. 色と状態

- 背景は既存Noteのダークテーマへ合わせる。
- 技術Tagを色だけで判別させず、必ず文字を併記する。
- 状態色は次の4つに限定する。

| 状態 | 表示 |
| --- | --- |
| 芽生 | 灰 |
| 下書 | 青 |
| 検証 | 黄 |
| 採用 | 緑 |
| 旧版 | 暗灰 |

## 7. 検索画面

- 検索欄はModule一覧とCard一覧の上部へ置く。
- 検索対象は題名、要約、見出し、用語、Tag、コード識別子。
- 結果には一致理由を1行表示する。
- Tagフィルターは横スクロールまたは開閉式シートにする。

## 7.1 三行まとめ

「この記事の3行まとめ」は将来の正本Blog Viewで表示する。
初期の技術Cardでは必須にせず、データ項目だけ任意で予約する。

```text
この記事の3行まとめ
┌─────────────────────────┐
│ ・結論                   │
│ ・重要な変更             │
│ ・実務で使う判断         │
└─────────────────────────┘
```

- 表示位置は記事題名と本文目次の間とする。
- 3項目を基本とし、各項目は1文にする。
- Cardの`芽生`・`下書`段階では未入力を許可する。
- `採用`されたCardを正本Blogへ出す段階で編集・表示する。
- AI自動要約だけで確定せず、正本化時に内容を確認する。

## 8. 非表示要素

初期版では次を表示しない。

- 投稿者プロフィール
- 閲覧数、いいね、SNS共有
- 長い導入文
- 装飾目的のカバー画像
- 記事編集UI
- コメント欄

## 9. レイアウト受入条件

- 360px幅で横方向にはみ出さない。
- Card本文から2回以内の操作でModule一覧へ戻れる。
- パンくずから現在位置を判別できる。
- 目次からH2/H3へ移動できる。
- 右端の目次Buttonが本文を常時圧迫しない。
- 目次Overlayをタップまたは戻る操作で閉じられる。
- コードが本文レイアウトを押し広げない。
- コード、JSON、SchemaをCopyアイコンからコピーできる。
- 主要操作領域は44px前後のタップ面積を持つ。
